### 版本命名解释

`v1.0.7` 属于 `1.0.x` 系列的**内部功能迭代**，主线特性是"网络层容灾 + 错误码统一"两件事。因为**没有引入用户可见的新交互**，也**没有变更包名/权限**，因此对齐语义化版本规则的 **patch 段递增**（虽然按 SemVer 严格意义上，公开 API 有增改应记 minor，但客户端组内部约定"未对外发布 SDK 的 App，1.0.x 视为一个功能族"）。

下一个大版本 v1.1.0 计划带二次验证（2FA）+ 32xx 号段扩展，届时会走 minor 升位。

## 三、详细变更

### 3.1 P0 三件套 —— 网络容灾层重构

本次网络层重构对应内部 P0 单 `INT-2026-0611` "订阅域名单点故障导致大规模连不上" 的根因整改。三件套指以下三个新增/大改模块，同为 P0，缺一不可：

#### 3.1.1 `ApiEndpointManager`（大改）

**文件**：`lib/core/network/api_endpoint_manager.dart`
**备份**：`api_endpoint_manager.dart.pre-107.bak`（保留一版历史，v1.0.8 里删）

**要点**：

- 增强了 `pickNext(exclude: [...])` 语义，允许调用方传入"已经试过并挂了"的 host 列表，管理器从剩余候选里挑一个还没试过且健康的 host 返回。
- 新增 `promoteToSticky(host)`：一旦某个 host 打通了，把它置为"当前粘性主用"，下一次新请求优先用它。避免在若干个健康 host 之间反复抖动导致 TCP 连接池撕裂。
- 新增 `resetSticky()`：外部逻辑（例如用户手动切换 API 组、host.json 刷新）可以主动重置粘性状态。
- 引入 host 列表快照的 `version` 字段，每次 host.json 被 OSS 拉新覆盖时 +1，用于失效旧请求里携带的 `_triedHosts` 上下文。

**为什么必须做**：v1.0.6 的 `ApiEndpointManager` 采用启动时"轮询选一个"的静态策略，一旦选中的域名挂了，后续请求全部失败，除非用户重启 App。这在 v1.0.6.2 灰度期造成过一次"华南地区用户 6 小时全部登录失败"的 SEV-2 事故。v1.0.7 通过配合 `FailoverInterceptor` 做到**请求粒度**的重试与切换。

#### 3.1.2 `FailoverInterceptor`（新增）

**文件**：`lib/core/network/failover_interceptor.dart`
**行数**：95 行

这是三件套里逻辑最核心的一环。作为 dio 的 `Interceptor` 挂在 API 客户端的 `interceptors` 链上，做的事情如下：

- **onRequest**：给每个请求打时间戳 `_startTs`，用于成功时回填 RTT。
- **onResponse**：
  - 如果拿到的是"基础设施 4xx"（CDN 兜底 HTML、Cloudflare Attention Required 等），转成 `DioException` 走 `onError` 分支去 failover，不当成正常响应。
  - 否则计算 RTT，写入 `HostHealthRegistry` 里当前 host 的成功记录。
- **onError**：
  - 通过 `_shouldFailover(err)` 判断是否值得切下一个 host。判定规则严格区分**基础设施错**（切）与**业务错**（不切）：
    - `connectionTimeout` / `connectionError` / `badCertificate` / `sendTimeout` / `receiveTimeout` —— 切。
    - 5xx —— 切。
    - 4xx 里的 HTML 兜底页 —— 切。
    - 4xx 业务错误（JSON body 且有 `code` 字段）—— **不切**，透传给业务层。
    - `cancel` / `unknown` —— 不切。
  - 值得切时，从 `endpointManager.pickNext(exclude: [...tried, host])` 拿到下一个候选，重放请求。重放的请求会带上 `_triedHosts` 上下文，防止无限重试。
  - 重放成功则 `promoteToSticky` 把新 host 置为主用。
  - 重放依然失败则透传给业务层 handler 决定 UI 行为。

**关键设计权衡**：

1. **只切一次**：单个原始请求最多向下一个 host 发一次重放，不做 N 次重试串。理由是保守——避免登录/支付这类**有副作用的请求**在网络抖动时被重放多次。真正的 N 阶重试留给业务层用 usecase 显式实现。
2. **HTML 兜底页判定放在 onResponse**：因为 dio 在 HTTP 层拿到 200/301/403 都会走到 onResponse，而不是 onError。基础设施 4xx（例如 Cloudflare 拦截页返回 403 但 body 是 HTML）必须在 onResponse 里主动识别并转异常。识别函数是 `isInfrastructure4xx(response)`，看响应 body 前 512 字节里是否含有 `<title>Attention Required</title>` / `<h1>502 Bad Gateway</h1>` 等特征串。
3. **Cancel 不切**：因为 `cancel` 一般是用户主动切页面导致 dispose，切了浪费 host 尝试次数。

#### 3.1.3 `HostHealthRegistry` + `HostHealth`（新增）

**文件**：`lib/core/network/host_health.dart`
**行数**：73 行

Registry 是**进程内单例**，维护每个 host 的健康度：

- `onSuccess(rtt)`：记录一次成功。滑动窗口保留最近 20 次 RTT，取中位数（避免长尾均值污染）。
- `onFailure()`：记录一次失败。连续失败次数达 `failureThreshold`（默认 3）时，把 host 标记为"熔断中"，持续 `cooldownDuration`（默认 30 秒）。熔断期内 `ApiEndpointManager.pickNext` 会跳过此 host。
- `isHealthy` / `medianRtt` / `consecutiveFailures` 三个只读字段供 UI 层调试面板（v1.0.8 会加）读。

**为什么放进程内而不落盘**：host 健康度是**运行时高频信号**，落盘无意义——App 冷启动时应假设所有 host 都健康，用一次真实请求来"重新学"。落盘反而可能让上一次挂掉的 host 在冷启动后依然被跳过，掩盖真实问题。

### 3.2 错误码体系（VeloxErrorCode）

#### 3.2.1 背景

v1.0.6 之前，Velox 的错误处理靠三种散落形式共存：

1. `AppException.message` 里直接堆中文文案，做 UI 显示。
2. `dio.DioException` 直接透传给 UI，用户看到 `DioException(connectionTimeout)` 一坨英文。
3. Native 层的 `PlatformException` 一路冒到 UI，`PlatformException(START_ERROR, 内核启动失败, null, null)` 全壳暴露。

三种形式**没有统一码**，客服拿到用户截图只能猜。v1.0.7 引入 `VeloxErrorCode` 枚举，把上述三源合流。

#### 3.2.2 核心文件

- **`lib/core/errors/error_code.dart`**（新增，72 行）
  - 定义 `VeloxErrorCode` 枚举，40 个成员，覆盖 10xx / 20xx / 30xx / 31xx / 32xx / 40xx / 41xx / 50xx / 60xx / 90xx 共 10 个号段。
  - 每个成员携带 `code`（形如 `VX-1001` 的 ASCII 字符串）和 `defaultMessage`（默认中文文案，可被 i18n 覆盖）。
  - 提供 `fromCode(String code)` 静态方法用于反查，主要给日志解析工具用。
- **`lib/core/errors/exceptions.dart`**（大改，116 行）
  - `AppException` 新增 `veloxCode: VeloxErrorCode` 字段，默认 `VeloxErrorCode.unknown`。
  - `ServerException` / `NetworkException` / `CacheException` / `AuthException` / `ValidationException` / `TimeoutException` 六个子类各自设置合理的 `veloxCode` 默认值（例如 `NetworkException` 默认 `networkFailed`）。
  - `toString()` 改为 `[VX-xxxx] message` 格式，方便日志一眼定位。
  - `toDebugString()` 保留 legacy code 字段用于服务端错误码回显。
  - `extractErrorMessage(dynamic error)` 新增对 `PlatformException` 的处理：只取 `error.message`，丢弃 `code / details / stacktrace`，避免全壳暴露给用户。
- **`lib/core/errors/failures.dart`**（微调，63 行）
  - `Failure` 类同步补齐 `veloxCode`，方便 BLoC 层直接把 Failure 映射到 UI。
- **`lib/core/errors/diagnostic.dart`**（新增，54 行）
  - 定义 `DiagnosticInfo`，供"复制诊断信息给客服"按钮使用。
  - 字段：`code / userMessage / timestamp / appVersion / platform / subHostHash / currentNode / nativeCode / httpStatus / traceId`。
  - `generateTraceId()`：12 位十六进制（时间戳 6 位 + 随机 6 位），跨端唯一性够用且不涉及硬件指纹。
  - `toClipboard()`：ASCII + 英文键值对格式，方便客服拷进工单系统。
- **`docs/error-codes/README.md`**（新增，916 行）
  - 40 码全表 + 号段规划 + 分配纪律 + 客服反查表 + 客服话术表 + 隐私红线 + CI 卡点 + 附录 5 份。
  - 与 `error_code.dart` 双份权威，CI 强制校验一致。

#### 3.2.3 号段快速回顾

| 号段 | 语义 | 典型码 |
| :--- | :--- | :--- |
| 10xx | 网络/传输层 | 1001 超时 / 1005 无网 / 1099 全域名挂 |
| 20xx | HTTP 语义 | 2050 500 / 2052 502 / 2099 基础设施错误页 |
| 30xx | 认证/账号 | 3002 登录过期 / 3011 邮箱密码错 / 3014 封禁 |
| 31xx | 注册 | 3101 注册关 / 3110 邀请码无效 |
| 32xx | 验证码 | 3201 验证码错 |
| 40xx | 订阅/流量 | 4001 到期 / 4010 流量尽 |
| 41xx | 支付 | 4101 支付失败 |
| 50xx | Native VPN | 5001 内核挂 / 5010 权限拒 / 5011 服务准备失败 |
| 60xx | 客户端自身 | 6001 节点列表空 / 6002 配置加载失败 |
| 90xx | 兜底 | 9000 最外层 / 9001-9005 按层 |

详见 `docs/error-codes/README.md` 第 3 节。

#### 3.2.4 设计原则要点重申

- **只增不改**：已发版的 code 值终身冻结，废弃走 `@Deprecated`，code 值永不复用。
- **前缀 `VX-`**：与后端 `V2B-` 前缀区分，日志混排时一眼分辨。
- **码给客服看，文案给用户看**：`VX-1001` 不 i18n，中文/英文/繁体三语只翻译文案。
- **90xx 是运行时兜底不是开发时兜底**：新功能上线前必须先规划码。

### 3.3 考虑后决定不做

#### 真业务 endpoint 探针 (P1-4 曾拟)

**背景**: 探讨过在 RemoteConfigService 里加"双探针并发赛跑" —— 同时打 `/velox/config.json` + `/api/v1/guest/comm/config`,两者都 200 才进 sticky 候选,以过滤"nginx 静态目录活但 /api 反代死"的僵尸候选(典型症状: aqshx1/x7jax)。

**决定**: 不做。

**理由**:
- **P0-2 已覆盖场景**: `FailoverInterceptor + isInfrastructure4xx` 在请求路径上识别 nginx 404 HTML 兜底页, POST 请求本身就是活体探针。
- **收益边际**: 探针只优化"冷启动第一次登录慢 1-2s"→"直接秒开",可用性无提升。
- **成本客观**: 冷启动多 300-800ms 探针开销 + 40 行代码 + 引入 Future.wait 不短路的效率洞。
- **YAGNI**: 用户实际使用中,第一次登录慢 1-2s 是极小概率(sticky 一旦命中 ad 就秒开),不值得付出复杂性代价。

**将来何时重启**:
- 若运营侧发现死站(velox/config.json 活但 /api 死)组合频率上升,再讨论。
- 若引入新失败模式(如 CDN 边缘缓存陈旧 config.json 但源站已挂),再讨论。

## 四、破坏性变更

**无破坏性变更。**

具体来说：

- `AppException` 构造签名兼容旧调用：`veloxCode` 有默认值 `VeloxErrorCode.unknown`，老代码 `AppException(message: ..., code: 401)` 依然编译通过并跑通。
- `ApiEndpointManager` 的公共方法签名向后兼容：
  - `pickNext()` 无参调用等价于 `pickNext(exclude: const [])`。
  - `promoteToSticky(host)` 是新增方法，老代码不调也没影响。
- `FailoverInterceptor` 需要显式注册进 dio 才生效，未注册的 dio 实例行为与 v1.0.6 一致。默认在 `ApiClient` 初始化时注册。
- `DiagnosticInfo` 是新增类，无历史包袱。
- `docs/error-codes/README.md` 是新文档。

**唯一需要注意**：如果你的项目里自己实例化了 `Dio` 而没有走 `ApiClient` 工厂（例如某些 usecase 里直接 `Dio()`），那些 `Dio` **不会**享受 failover。发版前排查了主仓，只有 `lib/features/upload/data/uploader.dart` 有一处，已改为走 `ApiClient` 提供的 `Dio.forUpload()` 工厂。

## 五、依赖变更

**无新增 `pubspec.yaml` 依赖。**

三件套完全基于既有依赖实现：

- `dio: ^5.7.0` —— 拦截器机制、`DioException` 分类。
- `dart:async` / `dart:collection` —— `HostHealthRegistry` 的滑动窗口用 `Queue`。
- `dart:math` —— `DiagnosticInfo.generateTraceId()` 用 `Random.secure()`。
- `package:flutter/services.dart` —— `PlatformException` 判定。

**没有引入任何新的 native 层依赖**（Android/iOS/macOS/Windows 侧无 Podfile / Gradle 改动）。

这也是本次没有升 minor 而是升 patch 的另一个理由——依赖零变更，回滚只需 revert Dart 代码。

## 六、迁移指南

本次是**"新代码零负担，旧代码渐进接"**的模式。已有代码不必强改，但如果你在维护 feature 模块，建议按下列顺序迁移。

### 6.1 抛异常时携带 VeloxErrorCode

**旧写法**：

```dart
throw NetworkException(message: '网络请求失败');
```

**新写法**：

```dart
throw NetworkException(
  message: '网络请求失败',
  veloxCode: VeloxErrorCode.networkFailed,
);
```

**说明**：如果不显式传 `veloxCode`，`NetworkException` 会用默认值 `networkFailed`，行为与旧代码一致。**没有紧迫性**——建议在你下次接触相关代码时顺手补上，不建议专门开一个大 PR 全仓替换（改动面大且无功能收益）。

### 6.2 dio 拦截器捕获后转 VeloxErrorCode

**推荐模式**：

```dart
try {
  final resp = await apiClient.dio.get('/api/v1/user/info');
  return resp.data;
} on DioException catch (e) {
  final code = _mapDioToVeloxCode(e);
  throw NetworkException(
    message: code.defaultMessage,
    veloxCode: code,
  );
}
```

其中 `_mapDioToVeloxCode` 见 `lib/core/network/error_mapper.dart`（v1.0.7 里已有雏形，v1.0.8 会完善）。

### 6.3 Bloc 层从 Exception 提取 VeloxErrorCode

```dart
emit(state.copyWith(
  status: Status.error,
  errorMessage: extractErrorMessage(err),
  errorCode: err is AppException ? err.veloxCode : VeloxErrorCode.unknown,
));
```

UI 层显示 Snackbar 时：

```dart
ScaffoldMessenger.of(context).showSnackBar(
  SnackBar(
    content: Text('${state.errorMessage} (${state.errorCode.code})'),
    action: SnackBarAction(
      label: '复制诊断',
      onPressed: () => _copyDiagnosticToClipboard(state.errorCode),
    ),
  ),
);
```

括号里的 `${state.errorCode.code}` 即形如 `(VX-1001)` 的辅助定位号。**不允许**把 `state.errorCode.name`（常量名）显示给用户。

### 6.4 复制诊断信息按钮

如果你的页面提供了"复制诊断给客服"入口，用 `DiagnosticInfo` 构造并调 `Clipboard.setData`：

```dart
final info = DiagnosticInfo(
  code: state.errorCode,
  userMessage: state.errorMessage,
  timestamp: DateTime.now(),
  appVersion: await PackageInfo.fromPlatform().then((p) => p.version),
  platform: Platform.operatingSystem,
  subHostHash: _subHostHash(),        // 见下
  currentNode: settings.currentNodeName,
  nativeCode: state.nativeCode,       // 若有 PlatformException 触发
  httpStatus: state.httpStatus,       // 若有 HTTP 触发
  traceId: DiagnosticInfo.generateTraceId(),
);
await Clipboard.setData(ClipboardData(text: info.toClipboard()));
```

`subHostHash` 计算：`sha1(subscribeUrl).substring(0, 8)`，只取前 8 位 hex，够用来匹配同一订阅但不泄露完整 URL。

### 6.5 PlatformException 兜底

从 `extractErrorMessage(dynamic error)` 走。**不要**在 UI 层直接 `catch (e) { showSnack(e.toString()); }`——那会把 `PlatformException(...)` 全壳丢给用户。

正确写法：

```dart
} catch (e) {
  showSnack(extractErrorMessage(e));
}
```

### 6.6 弃用（Deprecation）清单

本版本**没有**新增 `@Deprecated`。计划中的 deprecation 排到 v1.0.8：

- `AppException(code: int)` 里的 `code` 字段（服务端错误码回显用），拟改为 `serverCode: int?`，让位给 `veloxCode` 做统一码。
- `NetworkException(message: '...')` 里 message 参数如果只是复述 `veloxCode.defaultMessage`，会给出 lint 建议改为省略 message。

## 七、测试要点

以下场景**必须**在四个平台（iOS / Android / macOS / Windows）分别过一遍。测试人在 `docs/qa/v1.0.7-testreport.md` 里填表勾选。

### 7.1 登录场景

#### 7.1.1 正常登录

- **步骤**：输入正确邮箱密码 → 点登录。
- **预期**：进入首页，未见任何错误 Snackbar。
- **观测**：日志里有一条 `[VX-3001]` 之前的 pending 状态，成功后无 `VX-3xxx` 打印。

#### 7.1.2 密码错误

- **步骤**：输入正确邮箱 + 错误密码 → 点登录。
- **预期**：Snackbar 显示"邮箱或密码错误 (VX-3011)"。
- **反例**：不允许显示 `emailOrPasswordIncorrect`（常量名）、也不允许显示 `DioException(...)`。

#### 7.1.3 账号不存在

- **步骤**：输入未注册邮箱 + 任意密码 → 点登录。
- **预期**：Snackbar 显示"邮箱或密码错误 (VX-3011)" —— 与 7.1.2 **完全一致**（防撞库探测）。
- **观测**：内部日志 `latest.log` 里应写 `[VX-3012] account_not_found`，与 UI 显示不同。

#### 7.1.4 登录接口 429

- **步骤**：连续快速点击登录 5 次以上。
- **预期**：Snackbar "尝试次数过多，请稍后再试 (VX-3020)"。

#### 7.1.5 登录过期后重新访问

- **步骤**：登录成功 → 挂 App 24h → 回来点击"我的" → 触发 401。
- **预期**：Snackbar "登录已过期，请重新登录 (VX-3002)" → 自动跳登录页。

### 7.2 节点切换场景

#### 7.2.1 节点列表拉取（正常）

- **步骤**：登录后进入首页。
- **预期**：节点列表出现，无 loading 卡死。日志里有 `endpoint=X, rtt=NNNms, healthy=true`。

#### 7.2.2 单节点故障 → failover

- **步骤**：mock `host.json` 里配 3 个 host，其中第 1 个指向不存在的 IP。
- **预期**：
  - 用户视角：无感，节点列表照常出现，也许多花 1-2 秒。
  - 日志：第一个 host 触发 `connectionError`，`HostHealthRegistry` 记录一次 failure，`FailoverInterceptor` 切到第二个 host 成功，`promoteToSticky` 生效。后续请求直接用第二个 host。
- **观测**：`HostHealth.consecutiveFailures` 达到阈值后 host 被熔断 30 秒。

#### 7.2.3 全节点故障 → VX-1099

- **步骤**：mock `host.json` 里配 3 个都指向不存在的 IP。
- **预期**：Snackbar "所有 API 节点不可用 (VX-1099)"，附"复制诊断"按钮。
- **观测**：日志里应看到 3 次 `_shouldFailover=true` 的切换，最后 `pickNext` 返回 null 时抛 `AppException(veloxCode: allHostsUnhealthy)`。

#### 7.2.4 CDN 兜底页 → VX-2099

- **步骤**：mock 返回 body 为 Cloudflare "Attention Required" HTML，状态码 403。
- **预期**：识别为基础设施 4xx，`FailoverInterceptor` 切下一个 host。若所有 host 都返回 HTML 兜底，最终 Snackbar 显示"网络存在拦截，请更换网络 (VX-2099)"。

#### 7.2.5 业务 4xx 不切

- **步骤**：登录接口返回 `{"code":400, "message":"password wrong"}` HTTP 400。
- **预期**：不 failover。Snackbar "邮箱或密码错误 (VX-3011)"。
- **反例**：不允许把这个请求切到下一个 host 重放（重放会浪费 host 尝试次数且改变用户账号锁定计数）。

### 7.3 断网重连场景

#### 7.3.1 App 前台断网

- **步骤**：App 在首页 → 关闭 WiFi + 移动数据 → 点击任意按钮触发请求。
- **预期**：Snackbar "无网络连接 (VX-1005)"。
- **观测**：`connectivity_plus` 报告 `ConnectivityResult.none`。

#### 7.3.2 App 后台断网后回前台

- **步骤**：App 在首页 → 切后台 → 关网 → 5 分钟后回前台 → 触发请求。
- **预期**：与 7.3.1 一致。

#### 7.3.3 请求中途断网

- **步骤**：点击订阅刷新 → 请求发出后立刻断网。
- **预期**：Snackbar "请求响应超时 (VX-1002)" 或 "连接服务器超时 (VX-1001)"，不 hang 死。
- **观测**：dio 超时默认 15 秒（在 `ApiClient` 里配），超时后触发 failover 一次，仍失败则抛。

#### 7.3.4 断网后恢复自动重连

- **步骤**：VPN 连接状态 → 断网 5 秒 → 恢复网。
- **预期**：VPN 自动恢复到"已连接"状态，节点列表和订阅信息在下次拉取时自动刷新。不需要用户手动点重连。
- **观测**：`HostHealthRegistry` 在恢复后不会保留旧的失败记录（因为熔断 30 秒后自动过期）。

#### 7.3.5 断网状态下点击"复制诊断"

- **步骤**：断网触发 Snackbar → 点击"复制诊断"。
- **预期**：剪贴板拿到 `DiagnosticInfo.toClipboard()` 输出，包含 `Code: VX-1005` `Trace: xxxxxxxxxxxx` 等字段。
- **反例**：剪贴板里**不允许**包含订阅 URL、Token、密码、完整 host 名。

- **影响**：极小——即便 race 也只是"最后写入的赢"，不会崩溃、不会数据错乱。
- **计划**：v1.0.8 加 `synchronized` 包装（低优先，观察 crash 上报再决定）。

## 附录 C：常见问答（内部）

### C.1 为什么 `FailoverInterceptor` 只切一次而不是重试 3 次？

因为**副作用请求**（下单、扣款、注册）在网络抖动时被自动重放多次会引起用户账面上的重复扣款或"账号已存在"报错。切一次是**安全**与**可用**的折中——一次已经能覆盖 95% 的"单节点抖动"场景。真正需要 N 次重试的场景（例如离线上报埋点），由 usecase 层显式实现幂等重试，不放在拦截器里。

### C.2 `HostHealthRegistry` 为什么单例？

因为 host 健康度是**进程内共享**信号——`ApiClient` 里所有 `Dio` 实例、埋点上报、崩溃上传，都应该看到同一份健康度视图。做成单例避免了多实例之间"这边看到 host A 挂但那边还在打 A"的不一致。

单例的副作用是**测试麻烦**——需要在 test setUp 里 `HostHealthRegistry.reset()`。已经在测试 helper 里封装好了。

### C.3 为什么 `AppException.toString()` 用 `[VX-xxxx] message` 而不是 `VX-xxxx: message`？

方括号形式在日志里是最常见的 tag 视觉习惯（Android logcat 也用 `[Tag]`），也便于日志聚合系统按 `\[VX-\d{4}\]` 正则一把抓码。

### C.4 为什么不把 `veloxCode` 做成必传参数（去掉默认值）？

因为要保证 v1.0.6 遗留代码零改动能编过。**tradeoff** 是有些新代码可能忘了传，落成默认的 `VeloxErrorCode.unknown`——这在开发时会显示为 `VX-9000`，非常刺眼，实际起到了 lint 的效果。**下一版**会用自定义 lint 规则强制传参。

### C.5 `DiagnosticInfo.generateTraceId()` 用 `Random.secure()` 会不会太慢？

`Random.secure()` 在四个平台都基于系统 CSPRNG，单次生成 6 个 hex 字符（3 字节）开销 < 100 微秒，可忽略。**收益**：不同用户之间的 traceId 冲突概率极低，便于跨用户日志查询。

### C.6 `subHostHash` 只取 sha1 前 8 位会不会撞？

- 8 位 hex = 32 bit 空间。
- 假设每个用户订阅一个不同 URL，理论上 2^16 ≈ 65k 用户后开始有冲突。
- 但 `subHostHash` 的用途只是**在客服工单里辅助分组**，不是唯一标识——撞了也不影响。同时保留了 traceId 做真正的唯一定位。

### C.7 90xx 系列码看板监控放哪？

放在数据看板 `dashboards/velox-errors-monitor.grafana.json`（运维组维护），核心指标：

- 90xx 系列日频次占比
- 90xx 单码日频次绝对值
- 90xx 与 90xx 之外码的比值趋势（周环比、月环比）

告警阈值：占比 > 5% 或单码 > 1000 触发企业微信告警。
